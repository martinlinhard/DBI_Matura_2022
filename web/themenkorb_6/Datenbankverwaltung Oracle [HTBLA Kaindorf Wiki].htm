Datenbankverwaltung Oracle

Allgemeines

Datenbankimport

Als Datenbankimport bezeichnet man den Vorgang, wenn man Dateien wie

  • .csv
  • .txt
  • Excel-File
  • einem Datebankbackup

in die Datenbank importiert. Die Herausforderung dabei ist, die geforderten Daten, welche oft in einem schlechten Zustand (Fragmentierung, nicht normalisiert, NULL-Werte, falsche Daten, etc) vorliegen zu importieren. Bei diesem Vorgang helfen nun etwaige Tools wie der SQL-Developer, SQL-Loader oder je nach Zustand der Quelldaten auch Excel.

<font 10pt/Corbel,sans-serif;;inherit;;inherit>What do I have to consider when thinking of methods to import data?</font>

  • <font 10pt/Corbel,sans-serif;;inherit;;inherit>Quality of the data (not normalized, NULL-values, wrong data, etc…?)</font>
  • <font 10pt/Corbel,sans-serif;;inherit;;inherit>Amount of data (small enough to write INSERTs by hand? Big data amount?)</font>
  • <font 10pt/Corbel,sans-serif;;inherit;;inherit>Frequency of imports (How often does the import need to happen? Is it useful to automate the process?)</font>
  • <font 10pt/Corbel,sans-serif;;inherit;;inherit>Who will execute the import (Some employee of the company, a system engineer?)</font>

Datenbankexport

Ein Datenbankexport ist der Vorgang eine bereits vorhandene Datenbank in eine Datei(meist .sql - unterschiedlich je nach Export-Methode) zu exportieren. Der Vorteil dabei ist, dass diese abgespeichert und als Backup dienen kann.

SQL-Developer

Der SQL-Developer kann als Hilfestellung bei der Erstellung eines Control-Files zur Verwendung im SQL-Loader dienen. Wenn man bereits eine Datenbank im SQL-Developer erstellt hat, kann man über Rechtsklick im Kontextmenü, die Option „Import Data“ auswählen.

Danach öffnet sich das „Data Import“ Menü indem man die richtigen Delimiter und Optionen für die Verwendung in einem SQL-Loader einstellen.

Nach diesem Vorgang bestätigt man die Erstellung mit der Auswahl des SQL-Loaders im Kontextmenü.

Zulezt wählt man den Speicherort des Control-Files aus. Dieses File kann danach als Importhilfe im SQL-Loader benützt werden.

SQL-Loader

Allgemeine Funktionsweise

Der SQL-Loader ist ein sehr mächtiges Tool um Daten in eine Oracle-Datenbank zu importieren. Er verwendet Control-Dateien, welche beim Datenbankimport die Quelldateien und den Aufbau der Daten beschreiben.

Während des Imports werden erfolgreich importierte Daten in das Log-File geschrieben, Daten mit Fehlern in ein Bad-Log-File. Nach diesem Durchlauf können Daten aus dem Bad-Log-File nocheinmal angepasst und erneut importiert werden, ansonsten werden sie zu den Discarded-Files gegeben. Dieser Vorgang kann beliebig oft wiederholt werden bis alle Daten angepasst sind.

Begriffsübersicht

Die folgenden Kommandozeilen-Parameter können beim Start des Programms aufgerufen werden: 

userid=<connect_string>   Verbindung zur Datenbank
control=<steuerdatei>     der Name der Steuerdatei
log=<protokolldatei>      der Name der Protokolldatei
bad=<bad file>            der Name des bad file
data=<datendatei>         der Name der Datendatei
discard=<discard file>    der Name es discard file
discardmax=n              nach n Datensätzen im discard file wird abgebrochen
skip=n                    überlese die ersten n Datensätze (hilfreich zum Wiederaufsetzen)
load=n                    lade maximal n Datensätze
errors=n                  breche nach n fehlerhaften Datensätzen ab
direct=<modus>            Verwende den direkten Pfad (Modus=true) oder nicht (Modus=false)

Die Control-Datei

Die Control-Datei kann über mehrere Wege erstellt werden. Die erste Möglichkeit ist wie oben beschrieben über den SQL-Developer. Die zweite Methode ist das manuelle erstellen der Control-Datei. Die Control-Datei kann eine Reihe von Befehlskombinationen enthalten. Die Reihenfolge der Befehle ist relevant.

Kommentare

Kommentare können an beliebiger Stelle der Control-Datei eingebracht werden. Sie beginnen mit zwei waagerechten Bindestrichen und gehen bis zum Ende der Zeile.

Definition der Felder

Für jedes Feld muss der Spaltenname in der Datenbanktabelle, die Position innerhalb der Eingabedatei, der Datentyp und mögliche Feldende- oder Einschlusszeichen. (vgl. Einstellungen beim Import über den SQL-Developer)

Position innerhalb der Eingabedatei

Es gibt zwei Möglichkeiten, die Position eines Feldes in der Eingabedatei zu bestimmen. Bei der absoluten Positionierung wird die Start- und Endespalte innerhalb des logischen Datensatzes angegeben.

Beispiel:
auftrag_nr position(01) decimal external,
kunden_nr position(03:07) char  
1 17500 Maier
2 20340 Huber
Die angegebene auftrag_nr position(01) entspricht in diesem Fall 1 bei dem Kunden Maier mit der Kundennummer 17500 
kunden_nr position(03:07).

Bei der relativen Positionierung wird davon ausgegangen, dass alle Felder von links nach rechts angegeben werden.

Beispiel:
auftrag_nr decimal external,
kunden_nr char
Datentypen

Normalerweise ist es nicht nötig, die Datentypen der jeweiligen Felder anzugeben. Die Felder werden automatisch in die Typen der entsprechenden Tabellenspalten konvertiert. Eine Angabe der Datentypen muss nur bei binären Daten erfolgen. Die wesentlichen Datentypen sind „char“, „date“ und „Numeric EXTERNAL (INTEGER, FLOAT, DECIMAL)“. Bei Daten, die ein Datum repräsentieren, ist das Datumsformat anzugeben.

Beispiel:
auftrag_nr position(01) decimal external,
kunden_nr position(03:07) char,
datum position(08:17) date "dd.mm.yyyy"
Trennung von Feldern

Wenn die Feldpositionen relativ zu einander angegeben werden, muss klar sein, wo ein Feld endet und das nächste beginnt. Dazu wird entweder ein Endezeichen („terminated by“) oder Einschlusszeichen („enclosed by“) oder beides festgelegt.

Beispiel:
auftrag_nr terminated by whitespace,
kunden_nr terminated by ',',
datum enclosed by '(' and ')'

In den Beispiel wird die Auftragsnummer von einem Leerzeichen, Tabulatorzeichen oder dem Zeilenende terminiert (whitespace). Der Auftragsnummer folgt immer ein Komma und das Datum ist immer geklammert. Auch der optionale Einschluss ist möglich:

datum optionally enclosed by '(' and ')' Das Datum kann geklammert sein.

options

Bestimmte Einstellungen sind sowohl als Kommandozeilen-Parameter als auch in der Steuerdatei einstellbar.

Der Syntax in der Steuerdatei ist wie folgt: 

options (<paramter1>=wert1, <parameter2>=wert2, ... )
Beispiel:
options (rows = 1000, silent = (errors, discards))
unrecoverable - Laden ohne Redo

Einstellung in der Steuerdatei:

  • recoverable Redo-Informationen schreiben
  • unrecoverable keine Redo-Informationen schreiben

Der Ladevorgang mit der Einstellung „unrecoverable“ ist erheblich schneller. Sie bewirkt, dass über den Ladevorgang keine Redo-Informationen von der Datenbank geschrieben werden. Je nach Datenbank-Modus (Archive-Log-Mode) kann es passieren, dass die Daten in bestimmten Fehlerfällen verloren gehen können. Um dies zu verhindern, sollte nach dem Ladevorgang eine Komplettsicherung der Datenbank vorgenommen werden.

characterset - Einstellung des Zeichensatzes

Einstellung in der Steuerdatei: 

characterset <zeichensatz> --Zeichensatz einstellen

Damit Umlaute und Sonderzeichen korrekt umgesetzt werden, ist die Angabe des Zeichensatzes, in dem die externen Datendateien vorliegen, anzugeben. Für den Windows-Zeichensatzs ist dies WE8ISO8859P1, für DOS WE8PC850.

Beispiel:
characterset WE8ISO8859P1
Vorsicht:

Wenn der Zeichensatz nicht gesetzt ist, so wird die Definition der Umgebungsvariable NLS_LANG verwendet. Das direkte Laden setzt in diesem Fall jedoch voraus, dass die Daten in demselben Zeichensatz vorliegen, der dem Zeichensatz der Datenbank entspricht. Aus diesem Grund sollte der Zeichensatz immer angegeben werden.

insert, append, replace, truncate - Laden in nicht leere Tabellen

Einstellung in der Steuerdatei:

  • insert Laden nur in leere Tabellen möglich (default)
  • append Daten werden hinzugefügt
  • replace alle Daten der betroffenen Tabellen werden vorher gelöscht
  • truncate wie „replace“; es werden jedoch unbenutzte Speicherbereiche freigegeben (extents)
Vorsicht

Bei „replace“ und „truncate“ werden alle Daten der betreffenden Tabelle vorher gelöscht, unabhängig davon, welche Daten beim Ladevorgang hinzugefügt werden. Alle Daten der Tabelle sind unwiederbringlich verloren!

preserve blanks - Leerzeichen beibehalten

Einstellung in der Steuerdatei: preserve blanks Leerzeichen werden nicht abgeschnitten

Normalerweise werden bei festen Feldlängen führende und nachfolgende Leerzeichen abgeschnitten. Durch die Angabe von „preserve blanks“ lässt sich dieses verhindern.

infile, badfile, discardfile - Festlegung der Dateinamen

Einstellung in der Steuerdatei:

  • infile <dateiname> externe Eingabedatei
  • badfile <dateiname> Bad File
  • discardfile <dateiname> Discard File

Die Dateinamen sind in einfache Hochkommata (') einzuschließen.

Beispiel:
infile 'daten\test.dat'

Je nach Version der Oracle-Software kann es notwendig sein, das Zeichen Backslash (\) zu maskieren. infile, badfile, discardfile 

infile 'daten\\test.dat'
concatenate - Datensätze zusammenfassen

Einstellung in der Steuerdatei: 

concatenate <anzahl> --feste Anzahl von Zeilen zusammenfassen

Dieser Befehl bewirkt das eine definierte Anzahl von Zeilen in der Eingabedatei logisch zusammengefasst werden, weil in der Eingabedatei nicht alle Daten eines Datensatzes in einer Zeile plaziert werden können.

Beispiel:
concatenate 2
into table - Angabe der Zieltabelle

Einstellung in der Steuerdatei: into table <tabelle> –Zieltabelle in der die Daten abgelegt werden soll

Weitere Einstellungsmöglichkeiten je Tabelle sind „when“ und „trailing“.

when - Bedingtes Laden

Einstellung in der Steuerdatei: 

when <bedingung1> and <bedingung2> ... --Bedingung, ob die Daten in die Tabelle geladen werden sollen

Wenn es vom Inhalt der Daten abhängt, in welche Tabelle die Daten geschrieben werden sollen, so ist mit der when-Klausel die entsprechende Bedingung zu definieren. Auf diese Weise lassen sich Verteiler implementieren.

Beispiel
into table fertige_auftraege when status='F'
(<felddefinitionen>)
into table stornierte_auftraege when status='S'
(<felddefinitionen>)
trailing nullcols - Fehlende leere Endspalten

Einstellung in der Steuerdatei: 

trailing nullcols --Fehlende leere Endspalten ignorieren

Wenn in der Eingabedatei leere Felder am Ende der Datei fehlen, weil Leerzeichen am Ende einer Zeile entfernt wurden, verhindert „trailing nullcols“, dass diese Zeilen zu Fehlern führen. Die entsprechenden Spalten in der Datenbanktabelle erhalten dann den Wert NULL (leerer Wert). Dieses Vorgehen funktioniert jedoch nicht, bei absoluter Positionierung.

Weitere Tools

Oracle Import-Utility

Durch die Oracle Import-Utility lassen sich ausschließlich Oracle-Daten aus einer Exportdatei (dump) in eine Oracle-Datenbank importieren. Die Daten müssen vorher mit der Export-Utility exportiert worden sein. Die Parametrisierung kann entweder interaktiv oder Kommandozeilen orientiert erfolgen. Da bei der interaktiven Variante nicht sichergestellt ist, dass alle Parameter berücksichtigt werden, ist die Angabe der Parameter per Kommandozeile vorzuziehen.

Programmname: imp bzw. imp.exe

Data Pump

Oracle Data Pump ist eine neuere, schnellere und flexiblere Alternative im Verlgeich zu den klassichen „exp“ und „imp“ Funktionen der Oracle Versionen unter 10g. Zusätzlich zu der grundlegenden Import und Export Funktionalität wurde Data Pump um eine PL/SQL API erweitert und unterstützt den Import externer Tabellen.

Programmname: expdp bzw. impdp.exe
