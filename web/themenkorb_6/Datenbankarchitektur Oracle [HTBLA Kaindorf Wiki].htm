Datenbankarchitektur Oracle

Allgemeines

  • Instanz: Datenbankprozess
  • Datenbank: gespeicherte Daten
  • System Global Area(SGA): Jeder Prozess und Hintergrundprozess teilt sich eine SGA (mehrere Instanzen bedienen sich auch nur an einer SGA)
  • Program Global Areas (PGA): Jede Prozess hat einen privaten PGA, welchen er mit niemanden teilt.

System Global Area (SGA)

Die SGA besteht aus:

  • dem Database Buffer Cache
  • dem Shared Pool
  • dem Redo-Log Buffer
  • dem Data Dictionary Cache

Die System Global Area ist ein Speicherbereich, welcher für jede Oracle Instanz Daten und Kontrollinformationen bereitstellt. Wenn mehrere User in der gleichen Instanz gleichzeitig zugreifen, werden die Daten der SGA geteilt.

Komponenten der System Global Area

Database Buffer Cache

Der Database Buffer Cache speichert die zuletzt verwendeten Daten und Index Blocks um einen schnellen Zugriff gewährleisten zu können. Der Buffer besteht aus zwei Listen: Der „Write-List“(WL) und der „Least Recently Used List“ (LRU). Die WL enthält Daten, welche modifiziert aber noch nicht geschrieben wurden. Die LRU enthält Adressen von zuvor modifizierten oder freien Daten, welche lange nicht benützt wurden und wieder zum Beschreiben verfügbar sind. Der Database Buffer Cache ist im RAM.

Es gibt drei Arten von Blöcken:

  • Freie Blöcke (schwarz dargestellt)
  • Belegte Blöcke (weiß dargestellt)
  • Undo Blöcke für Rollback (grau dargestellt)

Shared Pool

Hier werden wichtige Daten von den schon ausgeführten SQL- und PL/SQL- Anweisungen und das Data Dictonary gespeichert. Dies dient dazu, um bei wiederholt gleichen Anfragen eine schnelle Antwort liefern zu können. Sobald der Shared Pool voll ist wird nach dem LRU-Prinzip (Least Recently Used) vorgegangen, was bedeutet, dass die am seltensten benutzten Anweisungen gelöscht werden.

Redo-Log Buffer

Der Redolog Buffer protokolliert bei jeder Änderung im Database Buffer Cache die Änderung mit. Diese Redo-Entries werden für die Wiederholung von Statements benötigt. Die geschriebenen Daten werden auf die essentiellen Informationen beschränkt.

Beispiele:

  • DELETE Primary Key, Tabellenname
  • INSERT kompletter Datensatz, Tabellenname

Der Redo-Log Buffer hat eine fixe Größe und ist dreigeteilt. Wenn das erste Drittel voll ist, wird im zweiten weitergeschrieben und der Log Writer (LGWR) beginnt das erste Drittel in die Online Redo-Log Dateien wegzuschreiben (Asynchron). Wenn das dritte Drittel voll ist, beginnt der Redo-Log Buffer wieder im ersten Drittel. Falls der Fall eintritt, dass die Daten vom ersten Drittel noch nicht weggeschrieben wurden, muss gewartet werden.

Data Dictionary Cache

Der Data Dictionary Cache ist eine der wichtigsten Speicherstrukturen in der Oracle Datenbankarchitektur. Er prüft beim Ausführen von SQL- und PL/SQL- Befehlen ab, ob die Spaltennamen existieren.

Prozesse

Schreiben von Daten

DataBase Writer (DBWR)

Der DBWR dient dazu, die Daten aus dem Database Buffer Cache in die Daten-Dateien zu schreiben. Dies macht er standardmäßig alle 3 Sekunden. Um die aktuell verwendeten Daten im Memory zu behalten, wird ein LRU-Algorithmus verwendet. So werden die am längsten nicht mehr verwendeten Blöcke weggeschrieben (müssen nicht die Blöcke sein, die sich am längsten im Buffer Cache befinden!). Das Wegschreiben passiert asynchron, was zur Vermeidung von Spitzen in der Auslastung beiträgt.

Der DBWR schreibt, wenn:

  • der Database Buffer Cache eine gewisse Größe erreicht
  • ein DBWR-Checkpoint erreicht wird
  • ein time-out auftritt
  • kein freier Buffer zu Verfügung steht
  • eine Liste zum Abarbeiten vom LGWR kommt

Log Writer (LGWR)

Im Redo Log Buffer sind alle Änderungen im Database Buffer Cache, diese schreibt der LGWR in die Redo Log Dateien. So kommen auch Undo-Informationen in die Redo Log Dateien. LGWR schreibt dabei sequentiell, d.h. hintereinander. Das geht, da die Redo Log Dateien eine fixe größe haben. Der Log Writer kann asynchron und synchron schreiben.

Bei jedem Commit schreibt der LGWR den gesamten Buffer syncrhon. Synchrones Schreiben ist beim LGWR kein Nachteil, da

* Teile davon bereits asynchron weggeschrieben wurden * Daten komprimiert sind (z.B. Tabelle und PK) * Der LGWR schneller schreibt (wird bei mehreren Benutzern laufend weggeschrieben) * Zusammenhängender Speicherplatz → schreibt sequentiell

Ein Commit ist erst dann fertig, wen der gesamte Redo Log Buffer leer ist. Auch bei Systemfehler können alle Daten wiederhergestellt werden, die vor dem Commit entstanden sind.

Log File Switch

Sobald eine Redolog Datei voll ist, wechselt der LGWR zu einer anderen Redolog Datei. Nun gibt es zwei Möglichkeiten, welche von der Konfiguration der Datenbank abhängig sind. Entweder die Datenbank läuft im archivelog Modus, was bedeutet, dass der ARCH Prozess die Daten der vollen Online Redolog Dateien in die Offline Dateien kopiert, oder sie läuft im noarchivelog Modus, bei welchem die Informationen überschrieben werden sobald der LGWR wieder zu dieser Redolog Datei kommt. Zuvor muss der LGWR dem DBWR sagen, dass er seine Strategie ändern soll und bestimmte Blöcke zuerst aus dem Database Buffer Cache wegschreibt, sodass die Logfile, die dann wieder überschrieben hat, nur Redo Informationen enthält, deren Blöcke schon auf den Daten Dateien abgeschrieben worden sind.

DataBase Writer (DBWR)

Ist ein Oracle Prozess, der im Hintergrund läuft, wenn eine Instanz der Datenbank gestartet wird. Er schreibt die Daten vom SGA in die Oracle Datenbank Dateien.

Check Point (CKPT)

Wenn der CKPT-Process aktiviert ist, übernimmt er die Aufgabe des LGWR, die Dateien an einem Checkpoint zu updaten. Am Ende eines Checkpoints wird der Header der Datafiles und Controlfiles upgedatet. Dies reduziert die Wiederherstellungszeit bei einem Fehler der Datenbank-Instanz. CKPT erhöht die Performance einer Datenbank mit vielen Database-Files.

Archiver (ARCH)

Wenn die Datenbank im Archive log mode ist und automatisches archivieren aktiviert ist, dann archiviert der Archiver automatisch Online Redo-Log Dateien und verhindert so, dass diese vom Log Writer überschrieben werden. Der Archiver archiviert diese Dateien als Offline Redo Log Dateien in ein Directory, dass vom User definiert wird.

Dafür wird der Parameter log_archive_start und log_archive_dest gesetzt.

* log_archive_start = true automatisches Archivieren * log_archive_dest = /oracle/arch/log archivelogdirectory definieren

System Monitor (SMON)

Der System Monitory wird automatisch gestartet, wenn eine neue Instanz von der Datenbank erstellt wird.

Er schreibt Redo Logs von Änderungen, die noch nicht auf die Festplatte geschrieben wurden, um im Falle eines Absturzes die Instanz wiederherstellen zu können. Außerdem führt er auch automatisch einen Rollback aus, wenn Transaktionen nicht commited wurden. Weiters werden auch alle 5min Wartungen druchgeführt, die freien Speicherplatz und temporäre Segmente freigeben.

Process Monitor (PMON)

Der Process Monitor ist zuständig für das process recovery, wenn ein User-Prozess versagt, oder vom Benutzer abgebrochen wird. Dieser reinigt dann den Cache und gibt Ressourcen, die vom Prozess genutzt wurden wieder frei.

Recoverer (RECO)

Der RECO ist dafür verantwortlich, Fehler, die in ausgelagerten Transaktionen aufgetreten sind, zu beheben.

Lock Prozess (LCKn)

Koordiniert das Sperren der Resourcen zwischen den verschiedenen Instanzen in einem Server.

Dedicated Server-Prozess

Der Dedicated Server Process fertigt genau einen User Process ab und nimmt dessen Statements entgegen. Dies ist sinnvoll, wenn es im System wenige User gibt, die sehr viel machen. Bei einem SELECT Statement holt der Dedicated Server die benötigten Daten entweder vom Database Buffer Cache oder von den Daten-Dateien.

Shared Server-Prozess

Der Shared Server Process kann mehrere User Processes abfertigen. Dies ist von Vorteil, wenn es viele User gibt, die alle wenig Operationen durchführen.

Dispatcher

Der Dispatcher sorgt für die Zuteilung zwischen User und Server.

Listener

Verbindet einen Client zum Datenbankserver.

Übersicht der Dateien

Die Datenbank Dateien speichern die Informationen der Datenbank. Bei Fehlern ermöglichen sie ein Recovery.

Daten Dateien

Diese enthalten die Benutzer- und Systemdaten der Datenbank.

Controldateien

Die Control Dateien enthalten eine Beschreibung der Datenbankstruktur und ermöglichen die Überprüfung und Sicherstellung der Integrität der Datenbank.

Redolog Dateien

Redolog Dateien

Die Redo Log Dateien beinhalten alle Befehle, welche der User eingegeben hat, in einer komprimierten Form. Alle Änderungen, welche Commited wurden, können mit ihrer Hilfe nachgestellt werden. Die Redo Log Dateien haben aber meist nur eine bestimmte Größe und werden mit der Zeit voll. Deshalb gibt es meist mehrere Platten zwischen denen der Logwriter beim beschreiben wechselt. Außerdem wird der Inhalt der vollen Platten vom Archiver auf die Offline Redo Log Dateien geschrieben, falls dieser aktiviert ist.

Offline Redolog Dateien

Die Offline Redolog Dateien sind ein längerfristiger Lagerplatz und meistens billiger als die Hardware für Redo Log Dateien, da sie nicht mehr so schnell arbeiten mussen.

Fehlerbehandlung

Lokaler Fehler

Bei lokalem Absturz wird ein Rollback mithilfe der Undo-Blöcke im Database Buffer Cache durchgeführt.

Rollback

Bei einem Stromausfall stellt der Systemmonitor (SMON) aus den Redo Log Datei wieder einen konstanten Zustand her.

Mediumsfehler

Bei einem Mediumsfehler (z.B. Brand) wird mittels Archive (ARCH) in den Offline Redo Log geschrieben. Bei dem nächsten bzw. übernächsten Backup wird der Offline Redo Log gelöscht/überschrieben.
