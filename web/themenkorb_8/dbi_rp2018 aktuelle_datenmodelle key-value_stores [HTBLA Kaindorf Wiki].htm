Redis

  • Entwickelt von Salvatore Sanfilippo, erschienen in 2009
  • In-Memory-Datenbank mit einer einfachen Schlüssel-Werte-Datenstruktur (Key-Value Store)
  • Laut db-engines.com Ranking ist Redis zurzeit die beliebteste Key-Value Store Datenbank

Speicherort: In-memory

Der Arbeitsspeicher eines Computers wird als Datenspeicher benutzt. Weil der Arbeitsspeicher eines Rechners wesentlich höhere Zugriffsgeschwindigkeiten als Festplattenlaufwerke hat, und die Algorithmen für den Zugriff einfacher sind, sind In-Memory-DB wesentlich schneller und ihre Zugriffszeiten sind besser vorhersagbar.
  • Vorteile: Performanter, schnellere Zugriffszeiten und besser vorhersehbare Zugriffszeiten.
  • Nachteile: Arbeitsspeicher ist teurer als Festplattenspeicher.

Primäres Datenbankmodell: Key-Value Store

Die wohl einfachste Form von DBMS. Es werden lediglich Paare von Schlüsseln und Werten abgespeichert, welche anhand des Schlüssels wieder zurückgeliefert werden können. Daher eignet sich dieses einfache System nicht für komplexe Applikationen. Key-Value-Stores kommen z.B. häufig ressourceschonend in eingebetteten System zum Einsatz, oder als In-Process-Datenbanken für besonders hohe Performance.
  • Vorteile: Besonders hohe Performance.
  • Nachteile: Nicht für komplexe Applikationen geeignet.

Lizenz: Open Source

Redis ist kein kommerzielles DBMS, es ist Open Source, also frei verwendbar.

Datenschema: Schemafrei

Ein Schema einer Datenbank definiert die Struktur der Daten, etwa Anzahl und Typ von Attributen einer Tabelle in einem relationalem DBMS. Ein DBMS, welche Schemen unterstützt, weist in der Regel Daten, die dieser Struktur nicht entsprechen, zurück. Schemafreie Datenbanken sind auch in der Lage schwächer strukturierte Daten abzuspeichern. Diese Art von DB bietet sich besonders dann an, wenn die Daten keine natürliche, einheitliche Struktur haben.
  • Vorteile: Flexibilität.
  • Nachteile: Verzicht auf Prüfung der Daten durch das DBMS.

Konsistenzkonzept: Eventual Consistency

Dieses Konsistenzkonzept ist eine abgeschwächte Variante der Konsistenz, die häufig bei verteilten Datenbanken zur Anwendung kommt. Dabei verzichtet man aus Performancegründen bei Schreiboperationen darauf, Daten sofort auf alle Server/Partitionen zu verteilen. Stattdessen kommen Algorithmen zum Einsatz, die sicherstellen, dass nach Beendigung der Schreiboperationen die Daten konsistent gemacht werden, ohne Aussage darüber, in welchem Zeitraum.
  • Vorteile: Bessere Performance
  • Nachteile: Das kann dazu führen, dass identische, zeitgleiche Abfragen mehrerer User unterschiedliche Ergebnisse liefern. Man kann lediglich darauf vertrauen, dass die Daten letztendlich konsistent sind.

Transaktionskonzept: Optimistic Locking

Verfahren, um in einem System parallelen Zugriff von mehreren Benutzern auf denselben Datensatz konfliktarm und ohne Inkonsistenzen zu regeln. Mehrere Nutzer bekommen den Parallelzugriff gewährt. Somit haben alle Leserechte. Wenn aber ein Datensatz von einem Benutzer geändert wurde, so bekommen andere Nutzer, die denselben Datensatz fast gleichzeitig zu ändern versuchen, eine Benachrichtigung, dass der Datensatz aktualisiert wurde.

Berechtigungskonzept: Passwort

Redis benutzt einen einfachen Passwort-basierten Zugriffsschutz.

Redis Sharding und Replication

Um eine Redis Datenbank auf mehreren Maschinen laufen zu lassen und die Daten auf diesen Maschinen abspeichern zu können, gibt es zwei nützliche Arten der Speicherung, die bei Redis benutzbar sind.

Beim Sharding, auch bekannt als Partitionierung, werden die Daten nach Keys aufgeteilt und auf den verschiedenen Partitionen/Maschinen gespeichert. Sharding ist nützlich, um die Performance zu verbessern und zu verhindern, dass eine Partition eine viel größere Datenmenge als die anderen speichern zu muss.

Beispiel: 
[1:Apple], [2:Banana], [3:Cherry], [4:Durian]
Zwei Maschinen, Maschine A und Maschine B
So könnten die vier Tupels mithilfe von Sharding abgespeichert werden:
2,4 auf Maschine A
1,3 auf Maschine B

Replikation, auch Spiegelung genannt, ist einfach das replizieren und speichern aller zu speichernden Daten auf allen Partitionen/Maschinen. Replikation ist nützlich, wenn man eine hohe Erhältlichkeit für Lesezugriffe braucht.

Beispiel: 
[1:Apple], [2:Banana], [3:Cherry], [4:Durian]
Zwei Maschinen, Maschine A und Maschine B
So werden die vier Tupels mithilfe von Replikation abgespeichert:
1, 2, 3, 4 auf Maschine A
1, 2, 3, 4 auf Maschine B

Anmerkungen

Der Name Redis entstand als Abkürzung für das englische „remote dictionary server“.

Die Persistenz wird durch automatisches, regelmäßiges Abspeichern oder per Protokolldatei geboten, wobei eine entsprechende Konfiguration auch eine ACID-konforme Dauerhaftigkeit erzeugen kann.

Redis online ausprobieren

Wichtige Befehle:

Beispiel

SchülerInnen-Daten: 
Klasse: 1AHIF
Katalognummer:1
Vorname: Max
Nachname: Musterschueler
Fehlstunden:0
Geburtdatum:17.1.2000

Klasse: 1BHIF
Katalognummer:3
Vorname: Maria
Nachname: Musterschuelerin
Fehlstunden:0
Geburtdatum:29.2.2000

a) Speichern der Werte (ohne Hashes) 
SET 1AHIF:1:Vorname Max
SET 1AHIF:1:Nachname Musterschueler
SET 1AHIF:1:Fehlstunden 0
SET 1AHIF:1:GebDat 17.01.2000

SET 1BHIF:3:Vorname Maria
SET 1BHIF:3:Nachname Musterschueler
SET 1BHIF:3:Fehlstunden 0
SET 1BHIF:3:GebDat 29.02.2000

b) Drei Abfragen: Formulierung der Aufgabenstellung + REDIS-Statement 
- Wie heißt Katalognummer 3 der 1BHIF mit Vornamen?
GET 1BHIF:3:Vorname
"Maria"

- Welches Geburtsdatum hat der Schüle mit Katalognummer 1 in der 1AHIF?
GET 1AHIF:1:GebDat
"17.1.2000"

- Gib die Fehlstunden der Schueler 1AHIF:1 und 1BHIF:3 aus.
MGET 1AHIF:1:Fehlstunden 1BHIF:3:Fehlstunden
1)	"0"
2)	"0"

a2) Speichern der Werte (mit Hashes) 
HMSET 1AHIF:1 Vorname "Max" Nachname "Musterschueler" Fehlstunden "0" Geburtsdatum "17.1.2000"

HMSET 1BHIF:3 Vorname "Maria" Nachname "Musterschuelerin" Fehlstunden "0" Geburtsdatum "29.2.2000"

b2) Drei Abfragen: Formulierung der Aufgabenstellung + REDIS-Statement 
-  Gib alle Informationen über den 1AHIF Schüler mit der Katalognummer 1 aus.
HVALS 1AHIF:1
1) "Maria"
2) "Musterschuelerin"
3) "0"
4) "29.2.2000"

- Existiert in der 1BHIF ein Schüler mit der Katalognummer 2?
EXISTS 1BHIF:2
(integer) 0

-  Wie heißt der Schüler mit der Katalognummer 1 aus der 1AHIF mit Nachnamen?
HGET 1AHIF:1 Nachname
"Musterschueler"
