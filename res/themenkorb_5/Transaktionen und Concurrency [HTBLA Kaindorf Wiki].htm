Transaktionen und Concurrency

Transaktionen

Allgemeines

Eine Transaktion ist eine Folge von Datenmanipulationsanweisungen (INSERT, UPDATE, DELETE, vgl. DML), die als atomare Einheit ausgeführt werden. Eine Transaktion überführt einen konsistenten DB-Zustand in einen wiederum konsistenten DB-Zustand und sie müssen nach dem ACID-Prinzip ablaufen. Eine Transaktion hat das alles oder nichts Prinzip, sie muss unteilbar sein.

Das ACID-Prinzip:

  1. Atomicity(Atomarität, Unteilbarkeit)
    • Veränderungen der Daten erfolgt vollständig oder gar nicht(alles oder nichts)
    • wird ein Einzelschritt nicht erfolgreich beendet, so wird der gesamte Vorgang zurückgesetzt
  2. Consistency(Konsistenz)
    • Daten sind immer in einem logischen konsistenten Zustand
    • Verarbeitung erfolgt von einem konsistenten Zustand zum nächsten
  3. Isolation
    • Datenänderungen innerhalb der Transaktion sind nur ihr selbst bekannt
    • Andere Transaktionen haben gleichzeitig keinen oder nur beschränkten Zugriff auf diese Daten
    • Transaktion darf keine andere, gleichzeitig mit ihr laufende beeinflussen
    • Problemberecih 'Synchronisation paralleler Transaktionen' oder auch Concurrency
  4. Durability(Persistenz, Dauerhaftigkeit)
    • Nach Transaktionsabschluss müssen die Datenänderungen dauerhaft (gegen Fehler jeglicher Art) bestehen bleiben
    • Problembereich 'Recovery'

Commit und Rollback

  • Mit einem COMMIT wird eine Transaktion ordnungsgemäß beendet. Somit werden die Veränderungen auch für andere Nutzer sichtbar.
  • Mit der ROLLBACK, Abort, Reset oder Blackout-Anweisung wird die gesamte Transaktion zurückgerollt, was heißt, dass alle durch DML-Anweisungen verursachten Datenänderungen rückgängig gemacht werden.

Autocommit

  • Der Autocommit-Modus ist der Standardmodus zur Transaktionsverwaltung. Für jede Transact-SQL-Anweisung wird beim Beenden ein Commit oder Rollback ausgeführt. Wird eine Anweisung erfolgreich beendet, so wird ein Commit ausgeführt. Wenn aber Fehler auftreten, wird ein Rollback ausgeführt. Eine Verbindung befindet sich immer dann im Autocommit-Modus, wenn dieser Standardmodus nicht durch explizite oder implizite Transaktionen außer Kraft gesetzt wird. Der Autocommit-Modus ist der Standardmodus für ADO, OLE DB, ODBC und DB-Library.
  • Eine Verbindung verbleibt im Autocommit-Modus, bis eine explizite Transaktion durch eine BEGIN TRANSACTION-Anweisung gestartet oder eine implizite Transaktion aktiviert wird. Wenn für die explizite Transaktion ein Commit oder Rollback ausgeführt wird oder wenn der implizite Transaktionsmodus deaktiviert wird, wechselt die Verbindung wieder in den Autocommit-Modus.
  • Beispiel:
  1. Ist SET IMPLICIT_TRANSACTIONS auf ON festgelegt, so wird für die Verbindung der implizite Transaktionsmodus festgelegt.
  2. Ist SET IMPLICIT_TRANSACTIONS auf OFF festgelegt, so wechselt die Verbindung wieder in den Autocommit-Transaktionsmodus.

Beginn und Ende einer Transaktion Im Anwendungsprogramm werden Beginn und Ende

  • entweder explizit angegeben (BEGIN/ COMMIT TRANSACTION, COMMIT WORK für Transaktionsende)
  • oder implizit angenommen (z.B. Prozeduren oder bestimmte Blöcke)

Länge von Transaktionen

  • so kurz als möglich:
    • geringer Systemoverhead
    • wenige Funktionen sind im Fehlerfall zu wiederholen
    • kurze Sperreinheiten für andere Anwender
  • so lang als notwendig:
    • konsistenter Datenzustand muss gewährleistet sein

Anomalien im Einbenutzerbetrieb

Eine Anomalie ist ein Fehlverhalten der Datenbank. Im Einbenutzer Betrieb können Anomalien durch nicht normalisierte bzw. denormalisierte Datenstrukturen entstehen und führen zu Inkontinenzen.

Probleme

Einfüge Anomalie

  • Man von einer Einfüge-Anomalie (Insertion-Anomalie) beim Einfügen von Daten in die DB, wenn eine neue Entität eines Entitätstyps in die Datenbank nicht oder nur schwierig eingetragen werden kann, weil nicht zu allen Attributen des Primärschlüssels Werte vorliegen. So können z.B. Informationen nicht aufgenommen werden, da in diesem Zusammenhang uninteressante bzw. zu diesem Zeitpunkt unbekannte Angaben fehlen.
  • Beispiel: In einer Tabelle wird für Fahrzeuge der jeweilige Fahrer angegeben. Die Attribute (Kennzeichen, Nachname) sind Identifikationsschlüssel. Hier treten Einfügeanomalien auf, wenn mehrere Fahrer mit gleichem Nachnamen dasselbe Fahrzeug fahren, oder für mehrere Fahrzeuge noch kein Fahrer bestimmt wurde.
  • Das Einfügen von Datensätzen mit gleichen Schlüsseln in eine Datenbank ist schwierig oder unmöglich.

Änderungs Anomalie

  • Beim Ändern von Daten in einer Datenbank spricht man von einer Änderungs-Anomalie (Update-Anomalie), wenn nicht alle Vorkommen einer Entität zugleich geändert werden. Dieses kann zu inkonsistenten Daten führen.
  • Beispiel: Es wird in einer Tabelle davon ausgegangen, dass die Erwähnungen von „Max Maier“ für ein und dieselbe Person gelten. Wird der Name „Maier“ in „Meier“ geändert, muss das an zwei Stellen geschehen. Geschieht das nicht, spricht man von einer Update-Anomalie. Die Tabelle enthält nun inkonsistente Daten.
  • Um dieses Problem zu verhindern, sollte die Tabelle in die 3. Normalform gebracht werden.

Lösch Anomalie

  • Eine Lösch-Anomalie (Delete-Anomalie) entsteht, wenn durch das Löschen eines Datensatzes mehr Informationen als erwünscht verloren gehen, wenn ein Datensatz mehrere unabhängige Informationen enthält. Durch das Löschen der einen Information wird dann auch die andere gelöscht, obwohl diese noch benötigt wird.
  • Beispiel: Wenn Kennzeichen, Auto und Name des Fahrers in einer Tabelle stehen, kann man das Fahrzeug nicht löschen ohne das man auch den Fahrer löscht.
  • Um das Problem zu vermeiden, muss die Tabelle in die 3. Normalform gebracht werden.

Concurrency (Synchronisation paralleler Transaktionen)

Eine der Haupteigenschaften von Datenbanksystemen ist, dass viele Benutzer gleichzeitig darauf zugreifen können, ohne dass die Konsistenz der Daten verletzt wird. (Concurrency-Control)

Bei unkontrollierten Zugriffen auf Datenobjekte im Mehrbenutzerbetrieb können einige unerwünschte Probleme auftreten, sogennante Anomalien. Auf diese Problemstellungen soll nun näher eingegangen und durch Beispiele verdeutlicht werden.

Problemstellung

  • Lost-Update Problem
  • Inconsistent Analysis Problem
  • Uncommitted Dependency Problem

Lost-Update Problem

Unter dem Lost-Update-Problem versteht man, dass eine Transaktion den veränderten Wert einer anderen Transaktion überschreibt, ohne zu erkennen, dass zwischen der eigenen Leseaktion und der anschließenden Änderung die gelesenen Daten von dieser anderen Transaktion geändert wurden.

Transaktion A Zeit Transaktion B
FETCH R t1 -
- t2 FETCH R
UPDATE R t3 -
- t4 UPDATE R

Problem: Änderung der Transaktion A zum Zeipunkt t3 wurden von Transaktion B zum Zeitpunkt t4 überschreiben und gehen somit verloren.

Inconsistent Analysis Problem

Die inkonsistente Analyse betrifft die Lesetransaktionen, die aufgrund parallel durchgeführter Änderungen während ihrer Ausführungszeit unterschiedliche Zustände der Datenbank sehen.

Gegeben: R1.stand=40 | R2.stand=50 | R3 stand=30
Tansaktion A –> Summiert die Kontostände auf
Transaktion B –> Bucht 10 Einheiten von R3.stand(-10) auf R1.stand(+10) um

Transaktion A Zeit Transaktion B
FETCH R1 t1 -
sum = sum + stand (sum:40)
FETCH R2 t2 -
sum = sum + stand (sum:90)
- t3 FETCH R3
stand = stand - 10
- t4 UPDATE R3 (stand: 20)
- t5 FETCH R1
stand = stand + 10
- t6 UPDATE R1 (stand: 50)
- t7 COMMIT
FETCH R3 t8 -
sum = sum + stand (sum: 110!)

Problem: Transaktion A ermittelt eine falsche Summe t8, richtig wäre 120.

Dirty-read

Aufgrund parallel durchgeführter Änderungen während der Ausführungszeit kommt es zu falsch ausgelesenen Werten.

Transaktion A Zeit Transaktion B
- t1 UPDATE R
FETCH R t2 -
- t3 ROLLBACK

Problem: Transaktion A liest Daten t2, die logisch nie existiert haben t3 weil die die Änderung nicht durchgeführt sonder zurückgesetzt wurde. Man könnte auch sagen, es wurde ein Phantom-Wert ausgelesen. Einen solchen Lesevorgang bezeichnet man als Dirty Read.

Transaktion A Zeit Transaktion B
- t1 UPDATE R
UPDATE R t2 -
- t3 ROLLBACK

Probleme:

  • Transaktion A ändert Daten (t2), ausgehen von einem nicht existierenden Stand
  • Änderungen werden wieder zurückgenommen (t3). Variante des Lost Update Problems

Serialisierbarkeit

Der Zustand der Datenbank muss während eines Lesezugriff nach dem Ausführen von konkirrierenden Transaktionen der selbe sein wie während/nach einem beliebigen seriellen (sequentiellen) Ablauf der Transaktionen.

Schedule

Um zu überprüfen, ob ein bestimmter Schedule, d.h. ein Ausführungsplan von Transaktionen mit ihren zugehörigen Operationen, serialisierbar ist, kann ein Serialisierbarkeits- oder Abhängigkeitsgraph darstellen. In diesem Graphen treten die einzelnen Transaktionen als Knoten auf und die Abhängigkeiten zwischen Transaktionen als (gerichtete) Kanten. Eine Abhängigkeit zwischen zwei Transaktionen Ti und Tj liegt vor, wenn Ti vor Tj auf dasselbe Objekt zugegriffen hat und die Operationen der beiden Transaktionen nicht reihenfolgeunabhängig (kommutativ) sind. Werden (wie üblich) als Operationen Lese- und Schreibzugriffe betrachtet, dann liegt eine Abhängigkeit vor, wenn wenigstens eine der beiden Operationen eine Schreiboperation darstellt. Es kann gezeigt werden, dass ein Schedule genau dann serialisierbar ist, wenn der zugehörige Abhängigkeitsgraph azyklisch ist. Denn nur in diesem Fall reflektiert der Graph eine partielle Ordnung zwischen den Transaktionen, die zu einer vollständigen Ordnung, die zugleich den äquivalenten seriellen Schedule bestimmt, erweitert werden kann.

Folgendes Beispiel zeigt eine Schedule mit drei Transaktionen, wobei r(x) bzw. w(x) den Lese- bzw. Schreibzugriff der jeweiligen Transaktion auf Objekt x kennzeichnen. Der zugehörige Abhängigkeitsgraph ist in Abb. 8-1b gezeigt, wobei die Kanten zusätzlich mit dem die Abhängigkeit verursachenden Objekt gekennzeichnet sind. Da der Graph keinen Zyklus enthält, ist der gezeigte Schedule serialisierbar. Die Serialisierungsreihenfolge lautet T3 < T1 < T2.

Lösungsmöglichkeiten

Sperrverfahren sind dadurch gekennzeichnet, dass vor dem Zugriff auf ein Objekt für die betreffende Transaktion eine Sperre erwirbt, deren Modus dem Zugriffswunsch entspricht.

Man unterscheidet:

  • Serieller Schedule: Transaktionen werden hintereinander ausgeführt(Reihenfolge der Transaktionen ist beliebig)
  • Serialisierbarer Schedule: Schedule ist äquivalent zu irgendeinem Seriellen Schedule
  • Zwei äquivalente Schedules: Zwei in Konflikt stehende Operationen werden jeweils in derselben Reihenfolge ausgeführt
  • Konfliktoperationen: - gehören zu unterschiedlichen Transaktionen - greifen aus dasselbe Datenobjekt zu - mindestens eine Operation ist eine Schreiboperation

Lösungen von Concurrency Problemen

Bei der Lösung von Concurrency-Problemen gibt es verschiedene Lösungsmethoden

  • Pessimistische Sperr-Verfahren: Annahme, dass Konflikte auftreten. Daher werden die Objekte von den Transaktionen gesperrt, um damit den Zugriff anderer Transaktionen auf diese Objekte für eine bestimmte (möglichst kurze) Zeit einzuschränken oder zu verhindern.
  • Optimistische Sperr-Verfahren: Annahme, dass keine Konflikte auftreten. drei Phasen: Lesephase - Validierungsphase - Schreibphase. Die Objekte werden möglichst lange nicht gesperrt, scheitert allerdings die Validierung, muss reagiert werden.
  • Zeitstempel-Verfahren: Jede Transaktion erählt eine Zeitmarke (Startzeitpunkt). Synchronisiert wird so, dass die Abareitung äquivalent zu einem seriellen Schedule ist. Ein Konflikt tritt auf, falls eine Transaktion z.B. eine Leseanforderung auf ein Objekt absetzt, das bereits von einer 'jüngeren' Transaktionverändert wurde. Konflikte werden durch einen Neustart einer betroffenen Transaktion gelöst. Diese Methode hat im Zusammenhang mit optimistischen Sperrverfahren und verteilten Datenbanken an Bedeutung gewonnen.

Sperrobjekte

* Hierarchy

  • Datenbank
  • Tabelle
  • Physischer Block / Seite
  • Zeile

Vorteil feiner Sperreinheiten: Höhere Parallelität zwischen Transaktionen ist möglich
Vorteil grober Sperreinheiten: Geringerer Verwaltungsaufwand für die Sperren ist notwendig

In der Praxis stellt Sperren auf Zeilenebene die feinste Sperrform dar.

Sperrmodi

  • X-Lock(Exklusive Sperre): Wenn eine Transaktion A das Objekt R mit X-Lock gesperrt hat, kann eine andere Transaktion B das Objekt R weder mit X-Lock noch mit S-Lock sperren. In diesem Fall muss Transaktion B solange warten, bis Transaktion A das Objekt R freigegeben hat.
  • S-Lock(Gemeinsame Sperre):Wenn eine Transaktion A das Objekt R mit S-Lock gesperrt hat, kann eine andere Transaktion B dasselbe Objekt zwar zusätzlich ebenfalls mit S-Lock, nicht aber mit X-Lock sperren. Im zweiten Fall muss Transaktion B solange warten, bis Transaktion A das Objekt R freigegeben hat. Transaktion A kann ein Objekt R, das ausschließlich von ihr mit S-Lock belegt ist, sehr wohl mit X-Lock sperren.

Kompatibilitätsmatirx

Transaktion A hat Objekt gesperrt mit
X-Lock S-Lock kein Lock
Transaktion B
kann Objekt
sperren mit 		X-Lock nein nein ja
S-Lock nein ja ja
kein Lock ja ja ja
  • Konkrete Systeme arbeiten mit:
    • nur einem Sperrmodus (X-Lock): geringer Verwaltungsaufwand
    • beiden Sperrmodi (S-Lock und X-Lock): höhere Prallelität

Für jedes System muss der Sperrmechanismus definiert werden:
Wodurch (explizit oder implizit) bei welchen Objekt welche Sperre angebracht wird und wodurch diese wieder aufgehoben werden.

Realisierung von Locking

In einer Tabelle werden pro Transaktion alle Sperren und Sperranforderungen gespeichert

  • Identifikation der Transaktion
  • Identifikation des Sperrobjekts (z.B. Tabellenname, Zeilennummer)
  • Art der Sperre (S, X)
  • Kennzeichen, ob Sperre oder Sperranforderung

Deadlocks

Ein Deadlock (Verklemmung) bezeichnet einen Zustand, bei dem ein oder mehrere Transaktionen auf die Möglichkeit warten, einen LOCK auf Datenbankobjekte abzusetzen und sich dabei gegenseitig behindern.

Die Transaktion kann von sich aus nicht erkennen, ob sie in eine Deadlock-Situation kommt oder sich in einer Deadlocksituation befindet.

Transaktion A Zeit Transaktion B
FETCH R t1 -
(S-Lock auf R)
- t2 FETCH R
(S-Lock auf R)
UPDATE R t3 -
(Request X-Lock auf R)
(warten) DISPLAY R
(warten) DISPLAY 'Weiter ja/nein?'
(warten) READ Antwort
(warten) t4 COMMIT
(warten) (Release S-Lock auf R)
(Resume UPDATE R) t5 -
(X-Lock auf R)

Problem: Beim Warten nach der UPDATE-Anforderung (t3) ist für die Transaktion A nicht ersichtlich, ob ein Deadlock vorliegt oder nicht.

Das Deadlock-Problem wird systemmäßig im Wesentlichen auf 2 Arten behandelt:

  • Deadlock-Vermeidung: Besteht bei einer Sperranforderung die Gefahr, dass ein Deadlock auftreten kann, wird die Transaktion abgebrochen. Alle Objekte, die die Transaktion will, werden zu Beginn der Transaktion gesperrt.\\Mit dem Zeitstempel-Verfahren ist ebenfalls eine Deadlock-Vermeidung möglich.
  • Deadlock-Erkennung:

Erkennung:
Zur Deadlock-Erkennung wir ein (gerichteter) Wartegraph geführt:

  • Knoten: Im System vorhandene Transaktionen
  • Kanten: Wenn Transaktion A ein Objekt sperren will (Request), Transaktion B aber eine entsprechende Sperre auf dieses Objekt aufrecht hält und Transaktion A daher warten muss, wird eine Kante von A nach B gezeichnet. Zur besseren Übersicht können die Kanten auch mit dem Namen des Objekts und der Art der gewünschten Sperre bezeichnet werden. z.B. R1(X), R3(S)
  • Eine Deadlock-Situation liegt genau dann vor, wenn im Wartegraph Zyklen (Kreise) enthalten sind. Der Wartegraph muss von Zeit zu Zeit (wann?) auf das Vorhandensein von Zyklen geprüft werden.

Beseitigung:

  • Eine an der Deadlocksituation beteiligte Transaktion ist als 'Opfer' auszuwählen und zurückzusetzen

Die Re-Read-Methode

Wird ein Datensatz in einem Multi-Tasking-Betrieb geändert wird das Re-Read-Verfahren(Read-Before-Write-Verfahren) verwendet, welches folgende Kriterien entspricht:

  • Es darf zu keinen Inkonsistenzen kommen
  • Die Objekte dürfen nicht zu lange gesperrt sein
  • Es darf keine Deadlock-Situation auftreten

Grafische Darstellung

Beschreibung:

  1. Datensatz nach Record-Old ohne Sperre einlesen
  2. Datensatz(oder Teile davon) aus Record-Old nach Record-Update kopieren
  3. Datensatz Record-Update(oder Teile davon) vom Benutzer ändern lassen
  4. Datensatz nach Record-Check mit X-Lock noch einmal einlesen(Re-Read)
  5. Ganzen Datensatz in Record-Old mit ganzem Datensatz in Record-Check vergleichen
  6. wenn gleich: dann Satz auf Grund von Record-Update ändern
    wenn ungleich: dann Meldung 'Satz wurde mittlerweile geändert' an den Benutzer

Sperrmodus U-Lock

Wird bei Lese-Operationen angegeben, wenn danach eine Änderungs-Operation beabsichtigt ist.

Vorteil:

  • geringere Deadlock-Gefahr als bei Lesen mit S-Lock
  • höhere Parallelität als bei Lesen mit X-Lock

Kompabilitätsmatrix:

X-Lock U-Lock S-Lock kein Lock
X-Lock nein nein nein ja
U-Lock nein nein ja ja
S-Lock nein ja ja ja
kein Lock ja ja ja ja

Isolation Levels

Definition: Ein Grad, zu dem die Datenkonsistenz eingehalten wird.

Es werden nicht für einzelne Operationen Sperrmodi angegeben, sondern für Transaktionen Isolation Levels definiert, die festlegen welche Serialisierungsprobleme auftreten dürfen und welche nicht.

3 Arten von Serialisierungsproblemen:

  • Dirty Read: Transaktion A liest Zeile, die von einer anderen Transaktion B geändert wurde. B führt ein Rollback durch und A arbeitet mit einer Zeile, die so nie existiert hat.
  • Nonrepeatable Read(Fuzzy Read): Transaktion A liest eine Zeile; eine andere Transkation B ändert diese Zeile. A liest diese Zeile nochmals und erhält eine andere Version derselben Zeile.
  • Phantoms: Transaktion A liest mehrere Zeilen, die einer bestimmten Bedingung genügen, eine andere Transaktion B fügt eine Zeile ein, die auch diese Bedingung erfüllt. A wiederholt die Abfrage und erhält ein anderes Ergebnis.

Folgende Tabelle gibt an, ob bei einem bestimmten Isolation Level ein bestimmtes Problem auftreten kann:

Dirty Read Nonrepeatable Read Phantoms
Read Uncommitted ja ja ja
Read Committed nein ja ja
Repeatable Read nein nein ja
Serializable nein nein nein

Read Uncommitted: kein Lock beim Lesen
Read Committed: S-Lock auf Zeile beim Lesen - nicht Two-Phase Locking
Repeatable Read: S-Lock auf Zeile beim Lesen bis Transaktionsende
Serializable: einfach - S-Lock auf Tabelle(n) beim Lesen bis Transaktionsende ODER komplex - Predicate Locking; Zugriffspfad für die angegebene Bedingung sperren; Sperren 'nicht existenter' Zeilen

Hierarchische Sperrverfahren

Sperren von verschiedenen Granulaten/ auf verschiedenen Hierarchen
(Datenbank – Tabelle – (Seite) – Zeile)

Problem: Wenn eine Transaktion ein Objekt sperren will, muss für alle untergeordneten Objekte überprüft werden, ob ein Sperrkonflikt auftritt.

Beim Sperren eines Objekts muss Top-Down bei allen in der Hierarchie übergeordneten Objekten eine geeignete Sperre erworben werden, das freigeben erfolgt Bottum-Up.

Arten von Intent Locks:

  • IS-Lock(Intent Shared Lock): alle untergeordneten Objekte werden mit S-Lock gesperrt
  • IX-Lock(Intent Exclusive Lock): alle untergeordneten Objekte werden mit X-Lock gesperrt
  • SIX-Lock(Shared Intent Exclusive Lock): sperrt das Objekt mit S-Lock und alle untergeordneten Objekte mit X-Lock

Stärke der Sperrmodi(links ist stärker):

Sperrstrategie

  • Bevor eine Transaktion ein Objekt mit S-Lock sperren kann, müssen alle seine übergeordneten Objekte Top-Down (von oben nach unten) mit IS-Lock oder stärker gesperrt werden
  • Bevor eine Transaktion ein Objekt mit X-Lock sperren kann, müssen alle seine übergeordneten Objekte Top-Down (von oben nach unten) mit IX-Lock oder stärker gesperrt werden
  • Die sperren werden mit Buttom-Up (von unten nach oben) freigegeben, dass heißt es wird die Sperre nicht freigegeben, wenn die Transaktion noch untergeordnete Objekte gesperrt hat.

Backup und Recovery

Moderne Datenbanksysteme sollen dafür sorgen, dass auch Daten im Fehlerfall verfügbar sind. Jedoch ist dafür nicht die Datenbank alleine verantwortlich. Deshalb ist es notwendig, ein Backup zu machen. Oft werden die Begriffe Backup und Recovery vertauscht, die Unterschiede sind jedoch einfach zu verstehen:

  • Backup: Ein Backup ist die Kopie von Daten einer Datenbank mit der man später Daten rekonstruieren kann.
  • Recovery: Recovery beschreibt alle Tätigkeiten, nach denen die Daten nach einem Fehlerfall wieder verwendbar sind, also einfach gesagt das Wiederherstellen von Daten.

Backup

Backup ist also der Prozess, in dem Daten gesichert werden mit der Absicht, diese bei einem Verlust wiederherzustellen. Diese Daten sind daher mindestens zwei mal gespeichert.

Arten von Backups

Es gibt verschiedene Arten von Backups und daher auch verschiedene Möglichkeiten, Backups zu kategorisieren.

Kategorie 1: Menge der Daten, Häufigkeit der Backups

Full Backup

Beim Full Backup werden die gesamten Daten gesichert.

  • Vorteil: Die Restore-Zeit ist gering und durch die redundante Datenspeicherung ist es sehr sicher.
  • Nachteil: Es wird viel Speicherplatz benötigt und das Backup zu machen braucht sehr lange.
Partial Backup

Beim Partial Backup werden nicht immer alle Daten gesichert, sondern nur die, die sich verändert haben.

  • Vorteil: Es wird wenig Speicherplatz gebraucht und die Zeit, die das Backup benötigt, ist gering.
  • Nachteil: Die Restore-Zeit ist sehr hoch und es gibt kaum Redundanzen, daher ist es nicht so sicher wie ein Full Backup.

Das Partial Backup kann weiter in zwei Unterarten geteilt werden:

  • Differential Backup: Hierbei werden nur die Daten gespeichert, die sich seit dem letzten Full Backup geändert haben.
    • Vorteil: Die Restore-Zeit ist gering und es ist sicherer, da eine gewisse Redundanz gewährleistet ist.
    • Nachteil: Es wird viel Speicherplatz benötigt und das Backup dauert länger.
  • Incremental Backup: Hierbei werden nur die Daten gespeichert, die sich seit dem letzten Partial Backup geändert haben.
    • Vorteil: Es wird wenig Speicherplatz benötigt und das Backup zu machen dauert nicht lange.
    • Nachteil: Ein Restore dauert länger und das Backup ist nicht sehr sicher, da es nur wenige Redundanzen gibt.

Kategorie 2: Zustand des Systems zum Zeitpunkt des Backups

Offline Backup

Das Offline Backup, auch Cold Backup genannt, wird ausgeführt, wenn die Datenbank offline ist. Ist dies der Fall können keine Datenbankänderungen durchgeführt werden und ein konsistenter und integer Zustand wird gewährleistet. Dafür kann man zu dieser Zeit nicht mit der Datenbank arbeiten, was heutzutage nicht mehr möglich ist.

Online Backup

Das Online Backup, auch Hot Backup genannt, wird bei laufendem Betrieb ausgeführt. Dabei ist zu beachten, dass während des Backups Änderungen gemacht werden könnten. Dies kann zu Inkonsistenzen führen, mit denen aber auch umgegangen werden muss. Deshalb muss von Beginn des Backups die Änderungen mitprotokolliert werden. Um einen konsistenten Zustand einzuspielen, müssen die Redo Logs ausgeführt werden.

Recovery

Recovery bezeichnet die Wiederherstellung eines Konsistenten Datenzustands nach einer Fehlersituation.

Arten von Fehlersituationen

Es gibt drei Arten von Fehlersituationen.

Transaktionsfehler (lokaler Fehler)

Eine Transaktion ist eine Arbeitseinheit, die entweder ganz oder garnicht durchgeführt werden muss oder darf. Sie führt die Daten von einem konsistenten Zustand in einen anderen konsistenten Zustand über.

Bei einem Transaktionsfehler wird eine Transaktion nicht ordnungsgemäß beendet. Die Daten befinden sich dann in einem inkonsistenten Zustand.

Was kann einen Transaktionsfehler auslösen ?
  • Laufzeitfehler im Anwenderprogramm (z.B. Division durch 0)
  • Endlosschleife im Anwenderprogramm
  • Deadlock
  • Abbruch des Anwenderprogramms durch den Systembediener
  • Time-Out des Anwenderprogramms (Überschreiten einer CPU-Zeitgrenze)
  • kein verfügbarer Plattenplatz bei einem Schreibbefehl im Anwenderprogramm
  • Befehl zum Zurücknehmen der Transaktion im Anwenderprogramm selbst
Welche Maßnahmen sind erforderlich?

Alle Änderungen, die die Transaktion bis zum Abbruch gemacht hat, müssen im laufenden Systembetrieb zurückgenommen werden. Dies nennt man Transaction Recovery, Transaction Rollback oder Dynamic Transaction Backout. Nur so kann man einen konsistenten Zustand gewährleisten. Man spricht hier von Backward Recovery.

Wer ist involviert?
  • Der User, der mit einem DML Befehl eine Transaktion auslöst
  • Der Transaktionsmonitor, der die Änderungen zurücknimmt, also ein Rollback auslöst
  • Der Systembetreuer, wenn er für den Transaktionsfehler verantwortlich ist indem er der Datenbank beispielsweise den Stecker zieht

Systemfehler (globaler Fehler, Soft Crash)

Ein Systemfehler tritt dann auf, wenn statt einer Transaktion mehere Transaktionen nicht ordnungsgemäß beendet werden können. Die Daten befinden sich dann in einem inkonsistenten Zustand.

Was kann einen Systemfehler auslösen ?
  • Stromausfall
  • Abbruch des Transaktionsmonitors durch den Systembediener
  • Fehler im Betriebssystem
  • Verlust von Hauptspeicherinhalten
Welche Maßnahmen sind erforderlich?

Alle Änderungen der Transaktionen, die zum Zeitpunkt des Systemfehlers aktiv (in flight) waren, müssen beim Systemwiederstart zurückgenommen werden. Dies nennt man Crash Recovery und zählt zum Backward Recovery.

Wer ist involviert?
  • Die User, die die Transaktionen ausgelöst haben.
  • Der Systemadministrator, der die Datenbank wieder hochfaren muss

Mediumfehler (Speicherfehler, Hard Crash)

Bei diesem Fehler sind die Daten physikalisch zerstört oder nicht mehr lesbar.

Was kann einen Mediumfehler auslösen ?
  • Irrtümliches Löschen von Daten
  • Irrtümliches Formatieren von Festplatten
  • Fehler in der Dateiverwaltung des Betriebssystems
  • Katastrophenfälle (Brand, Erdbeben, etc.)
  • Hard-Crash auf der Magnetplatte
  • Fehler im Disk-Controller
Welche Maßnahmen sind erforderlich?

Ein Sicherungsstand (Backup Copy) der Daten wird eingespielt und die Änderungen aller Transaktionen, die seit dem Zeitpunkt der Sicherung beendet wurden, müssen vom System nachvollzogen werden. Dies nennt man Archive Recovery, Media Recovery oder Disaster Recovery, es zählt zum Forward Recovery.

Wer ist involviert?
  • Der Administrator, der neue Hardware kaufen muss und das Backup einspielen muss

Techniken für Recovery

Für die oben genannten Fehlersituationen gibt es zwei Techniken. Transaction Recovery und Crash Recovery werden unter demBegriff Backward Recovery zusammengefasst. Archive Recovery wird auch Forward Recovery genannt.

Bei kombinierten System- und Mediumfehlern ist Forward- und Backward-Recovery (in dieser Reihenfolge) durchzuführen. Bei gewissen Strategien werden die After-Images erst bei Transaktionsende auf die Log-Datei geschrieben. Dadurch kann das Backward-Recovery entfallen.

Backward Recovery

Logging

Hier spricht man auch noch von Journaling in Undo-Logs. Vor jeder Veränderung wird eine Kopie dieser Daten erstellt (=Before Image) und in die Undo-Log-Datei eingetragen. Diese Kopie ist das Before Image (Daten vor der Änderung). Dieses Prinzip nennt man Write Ahead Log (WAL). Durch diesen Vorgang ist für jede Operation wie aufnehmen, ändern und löschen der alte Wert bekannt. Beim Rückgängigmachen der Transaktion wird das Undo-Log rückwärts durchlaufen und jeweils der alte Wert wiederhergestellt.

In einer Undo-Log-Datei müssen folgende Informationen enthalten sein:

  • Identifikation der Transaktion
  • Kennzeichen der Operation: Welche Transaktion? Welche Tabelle? Welche Zeile?
  • Before-Image der Daten
  • Identifikation des Benutzers
  • Identifikation des Terminals
  • Identifikation des Programmes
  • Zeitstempel

Eigentlich müsste diese Log-Datei nur bis zum erfolgreichen Ende einer Transaktion aufgehoben werden, meistens wird sie aber zu Recovery-Zwecken fortlaufend geführt. Beim Transaction Recovery muss das Undo Log bis zum Beginn der Transaktion gelesen werden.

Beim Crash Recovery mus das Undo-Log bis zum Anfang durchgelesen werden, den es könnten imer noch Before-Images einer nicht beendeten Transaktion existieren.Um den Abschnitt zu begrenzen, in dem gesucht werden muss, verwendet man Checkpoints. Dabei werden die Transaktionen, die zu diesem Zeitpunkt aktiv sind, gespeichert.

Das Log-File wird rückwärts bis zum jüngsten Checkpoint gelesen. Wird zu einer Transaktion keine Endmarke gefunden, wird diese mittels des Before-Images rückgängig gemacht. Auch die Transaktionen die im Checkpoint angeführt sind und keine Endmarke besitzen werden rückgängig gemacht.

Verzögertes Update

Beim Verzögerte Update (Deferred Update) werden die Änderungen der Daten erst geschrieben, wenn die Transaktion erfolgreich beendet ist. Bis zu diesem Zeitpunkt wird auf Kopien der Daten gearbeitet. Beim Rollback werden einfach diese Kopien gelöscht.

Forward Recovery

ANMERKUN: Redo-Logs und die Spiegelplatten müssen beim Forward Recovery natürlich auf physisch anderen Disk-Drives als die Basis-Daten gehalten werden. Die Redo-Logs und Undo-Logs können gemeinsam gespeichert werden, da das After-Image gleich dem Before-Image der nächsten Änderung ist. Diese gemeinsamen Log-Dateien werden Audit Trail genannt.

Das Forward Recovery bietet zwei Strategien:

Logging

Die Log-Datei im Forward Recovery bezeichnet man Redo-Log. Hier werden die After-Images (Kopie der Daten nach der Änderung) in ein Redo-Log eingetragen. Dieses wird vom Anfang beginnend vorwärts gelesen.

Eine Redo-Log-Datei muss dieselben Informationen wie eine Undo-Log-Datei enthalten, jedoch anstatt eines Before-Images ein After-Image.

Redo-Informationen müssen bis zur erfolgreichen Erstellung des nächsten Backups aufgehoben werden.

Gespiegelte Platten

Spiegelung bietet eine Alternative zum Archive Recovery. Dies kann z.B. mit RAID umgesetzt werden. Die Daten werden redundant gehalten und laufend parallel geändert. Diese Technologien bieten einen schnellen Wiederanlauf und zusätzlich noch Hochverfügbarkeit. Dazu sind weniger Backup-Kopien notwendig. Die Hardwarekosten steigen jedoch und ausgelagerte Backups und Logs müssen für Katastrophenfälle trotzdem verwendet werden.

Data Control Language

Die Data Control Language (DCL) ist der Teil einer Datenbanksprache, der verwendet wird um Berechtigungen zu vergeben oder zu entziehen. Es ist die Datenüberwachungssprache einer Datenbank.

DCL für Sql

  • Neben DDL und DML liegt sie in Form von Englischen Befehlsklauseln vor. Beispiele:
    1. GRANT SELECT
    2. UPDATE ON INVOICE TO CLERK
    3. REVOKE EXECUTE ON NIGHTLY_JOB FROM DEVELOPERS

-Beispiele:

  • GRANT Operation+ ON Relation TO (PUBLIC|Benutzer) [WITH GRANT OPTION]
  • REVOKE Operation+ ON Relation FROM (PUBLIC|Benutzer)
  • Erklärung: Relation kann auch eine Sicht sein. WITH GRANT OPTION erlaubt es den neuen Rechteinhabern, das Recht weiterzugeben. PUBLIC bezeichnet alle Benutzer. Der Datenbankadministrator hat alle Rechte. Der Besitzer eines Objektes hat ebenfalls alle Rechte an diesem Objekt.
